[1m  term_to_binary(Term)[0m

  Returns a binary data object that is the result of encoding [4mTerm[0m
  according to the Erlang external term format.

  This can be used for various purposes, such as efficiently writing
  a term to a file or sending an Erlang term through a communication
  channel not supported by distributed Erlang.

  See also [4mbinary_to_term/1[0m.

  [4mNote[0m

    There is no guarantee that this function will always return
    the same encoded representation for the same term.

[1mExamples[0m

    1> Bin = term_to_binary(hello).
    <<131,119,5,104,101,108,108,111>>
    2> hello = binary_to_term(Bin).
    hello

[1m  term_to_binary(Term, Options)[0m

  Returns a binary data object that is the result of encoding [4mTerm[0m
  according to the Erlang external term format.

  Supported options:

   • [4mcompressed[0m - Compress the external term format. The
     compressed format is automatically recognized by [4m[0m
     [4mbinary_to_term/1[0m as from Erlang/OTP R7B.

   • [4m{compressed, Level}[0m - Compress the external term format to
     a given level. The compression level is specified by [4mLevel[0m
     which is an integer in the range 0 to 9, where:

      ￮ [4m0[0m - No compression is applied (equivalent to omitting
        the [4mcompressed[0m option).

      ￮ [4m1[0m - Fastest compression but may not compress as well
        as the higher levels.

      ￮ [4m6[0m - Default level when the [4mcompressed[0m option is
        provided.

      ￮ [4m9[0m - Highest compression level, taking the longest
        time while attempting to produce the smallest result.
        However, depending on the input term, level 9
        compression may not always produce a smaller result
        than level 1 compression.

   • [4m{minor_version, Version}[0m(Since R11B-4) The option can be
     used to control some encoding details. Valid values for [4m[0m
     [4mVersion[0m are:

      ￮ [4m0[0m - Floats are encoded using a textual
        representation.

        Atoms that can be represented by a latin1 string are
        encoded using latin1 while only atoms that cannot be
        represented by latin1 are encoded using utf8.

      ￮ [4m1[0m - Floats are encoded in a more space-efficient and
        exact way (namely in the 64-bit IEEE format, rather
        than converted to a textual representation). As from
        Erlang/OTP R11B-4, [4mbinary_to_term/1[0m can decode this
        representation.

        Atoms that can be represented by a latin1 string are
        encoded using latin1 while only atoms that cannot be
        represented by latin1 are encoded using utf8.

      ￮ [4m2[0m - This is as of Erlang/OTP 26.0 the default.
        Atoms are unconditionally encoded using utf8.
        Erlang/OTP systems as of R16B can decode this
        representation.

   • [4mdeterministic[0m(Since OTP 24.1) This option can be used to
     ensure that, within the same major release of Erlang/OTP,
     the same encoded representation is returned for the same
     term. There is still no guarantee that the encoded
     representation remains the same between major releases of
     Erlang/OTP.

     This option cannot be combined with the [4mlocal[0m option.

   • [4mlocal[0m (Since OTP 26.0) This option encodes [4mTerm[0m in an
     alternative local version of the external term format. When
     decoded by the same runtime system instance, it produces a
     term identical to original term, even if the node name
     and/or creation of the runtime system instance have changed
     between encoding and decoding.

     When encoding without the [4mlocal[0m option, local identifiers
     such as pids, ports, and references will not remain the same
     if node name and/or creation of the runtime system instance
     changed between encoding and decoding. This is because such
     identifiers refer to a specific node by node name and
     creation.

     The node name and creation change when the distribution is
     started or stopped. The distribution starts when the runtime
     system is started with the [4m-name[0m or [4m-sname[0m command-line
     arguments. Note that the actual start of the distribution
     occurs after other code in the startup phase has already
     begun executing. The distribution can also be started by
     calling [4mnet_kernel:start/2[0m and stopped by calling [4m[0m
     [4mnet_kernel:stop/1[0m, provided it was not started via the
     command line.

     When decoding a term encoded with the [4mlocal[0m option using,
     for example, [4mbinary_to_term/1[0m, the runtime system attempts
     to verify that the term was encoded by the same runtime
     system instance. In most cases, decoding will fail if it was
     encoded by a different instance. However, this verification
     is not foolproof. You should ensure that terms encoded
     with the local option are only decoded by the same Erlang
     runtime system instance that encoded them.

     Since only the runtime system that encoded a term using the [4m[0m
     [4mlocal[0m option can decode it, the local encoding is typically
     pieced together with something else to produce a reply to
     where the local encoding originates. If a term encoded with
     the local option has its leading version number stripped, it
     can be embedded as part of a larger term (for example as an
     element in a tuple) when encoding in the external term
     format using, for example, EI. In this case, you would strip
     the version number using [4mei_decode_version()[0m and append
     the remaining local encoding using, for example, [4m[0m
     [4mei_x_append_buf()[0m.

     A common use case for the [4mlocal[0m option is when making a
     request from a process to a port driver driver while
     leveraging the selective receive optimization for handling
     the reply.

     In this scenario:

      ￮ A reference is created.

      ￮ The reference is serialized using the external term
        format with the [4mlocal[0m option.

      ￮ This serialized reference is passed to the driver in
        the request.

      ￮ The process then waits for a reply message in a
        selective receive, matching on the reference.

     The driver should send the reply using either [4m[0m
     [4merl_drv_output_term()[0m or [4merl_drv_send_term()[0m using the
     term type [4mERL_DRV_EXT2TERM[0m for the reference previously
     received in the request.

     Note that you should not strip the leading version number
     from the local encoding when using the term type [4m[0m
     [4mERL_DRV_EXT2TERM[0m. If the reference is not encoded with the [4m[0m
     [4mlocal[0m option, and distribution is started or stopped while
     the request is ongoing, the requesting process will hang
     indefinitely because the reference in the reply message will
     never match.

     This option cannot be combined with the [4mdeterministic[0m
     option.

     For more details, see [4mLOCAL_EXT[0m.

     See also [4mbinary_to_term/1[0m.

[1mExamples[0m

    1> List = lists:duplicate(20, $=).
    "===================="
    2> term_to_binary(List, []).
    <<131,107,0,20,61,61,61,61,61,61,61,61,61,61,61,61,61,61,
      61,61,61,61,61,61>>
    3> term_to_binary(List, [compressed]).
    <<131,80,0,0,0,23,120,156,203,102,16,177,197,2,0,61,98,5,
      68>>
