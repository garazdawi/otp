## 1. Goals and non-goals

### Goals

- Add a *second tier* of JIT compilation, sitting **above BeamAsm**, on
  **aarch64** (Apple Silicon + Linux ARM64 first; x86_64 later).
- Apply roughly the same SSA-level optimizations the AOT compiler does
  in `beam_ssa_opt` — but starting from **profile-narrowed type
  information** instead of `any`-typed function parameters.
- Perform **inlining of small functions** at SSA level, monomorphic
  call sites first, polymorphic later.
- **Coexist** with BeamAsm. T2 is opt-in, additive, and can be turned
  off without affecting correctness or T1 performance.
- **Side-exit to BeamAsm-compiled code** when speculations fail.
  Never bail to an interpreter (we don't have one on aarch64).
- Maintain the **single Erlang stack** that BeamAsm uses. No dual
  stacks, no separate calling convention.
- Keep the implementation **small** — target < 30 KLOC in C++ and Erlang
  combined, reusing as much existing infrastructure as possible.

### Non-goals

- A "FTL-equivalent" T3 with whole-program LLVM-quality optimization.
  T2 is the only new tier; T3 is explicitly future work.
- Tracing JIT semantics. We are method-based.
- Replacing BeamAsm. T1 stays the default and the source of truth for
  every BEAM op's semantics.
- **No** polymorphic specialisation in v1. Monomorphic call sites
  only at first; polymorphic via PIC-style or class-switch is roadmap.
  (Note: inlining a higher-order helper like `lists:foldl/3` with a
  literal-fun argument is *not* polymorphic — the fun is statically
  known via constant propagation. That case *is* in v1; see §9.1.)
- General OSR entry mid-loop in v1. Tier-up is per-call only initially.
  OSR entry is roadmap.
- x86_64 in v1. The architecture should not preclude it, but the first
  target is aarch64 only.
- Cross-module deep optimization that requires invalidation broader
  than what code purge already handles (i.e. T2 dependency tracking
  is granular per-blob, not whole-system).

## 2. Why T2

The current BeamAsm JIT is a template-style code generator with no
inter-instruction optimization. It pays its way (full coverage, fast
load-time compilation, ARM64 + x86_64 support, no mode-mixing
complexity), and per "The Road to the JIT" it was a deliberate choice
to not chase peak throughput in the baseline tier.

But there are wins BeamAsm cannot reach on its own:

1. **Type-driven specialization beyond what AOT can prove.** The AOT
   compiler must treat exported function parameters as `any`. In a
   running system, almost every hot function is called with a stable,
   narrow type. T2 observes that and re-optimizes from a much
   more precise starting point.
2. **Inlining of small functions.** The AOT compiler does limited
   inlining at the Core Erlang level. SSA-level inlining with
   profile-guided receiver-class information is exactly what makes
   accessor chains and small wrappers go fast — which is where Erlang
   programmers naturally compose code.
3. **Cross-call constant folding.** Once we inline, constant folding,
   CSE, and dead-code elimination cascade across what used to be
   separate function compilations.
4. **Better register allocation across optimization regions.** BeamAsm
   has fixed X→CPU-register assignments for the first 6 X registers;
   T2 can re-allocate temporaries, defer X-register flushes, and keep
   intermediate values in untagged form.
5. **Specialised arithmetic and unboxed floats.** The AOT compiler's
   `ssa_opt_float` pass already starts this; T2 with profile-narrowed
   types extends the regions where float arithmetic stays unboxed
   across function calls (because we inlined them).

The case is *not* "BeamAsm is bad". The case is "BeamAsm did the right
thing for the baseline tier; the wins above need a second tier to
unlock".

## 3. Architecture in one diagram

```
                ┌──────────────────────────────────────────────────┐
                │  Erlang source                                   │
                │   ↓ AOT compiler                                 │
                │  BEAM file (with type info from beam_ssa_type)   │
                └──────────────────────┬───────────────────────────┘
                                       │ load-time
                                       ▼
        ┌──────────────────────────────────────────────────────────┐
        │  BeamAsm (T1) — template JIT                             │
        │   - per-instruction emitters                             │
        │   - asmjit codegen                                       │
        │   - exports addressv[active_code_ix] dispatch            │
        │   - **NEW**: per-function entry counter (cheap)          │
        │   - **NEW**: per-call-site monomorphic-target slot       │
        └──────────────────────┬───────────────────────────────────┘
                               │ counter trips
                               ▼
        ┌──────────────────────────────────────────────────────────┐
        │  T2 manager (Erlang process)                             │
        │   - reads BEAM SSA + profile data                        │
        │   - asks T2 optimizer to compile this {Mod,Fun,Arity}    │
        └──────────────────────┬───────────────────────────────────┘
                               │
                               ▼
        ┌──────────────────────────────────────────────────────────┐
        │  T2 optimizer (Erlang code, in lib/compiler or new app)  │
        │   1. Load function's BEAM SSA from BEAM file             │
        │   2. Narrow arg types from profile data                  │
        │   3. Insert speculation guards / deopt points            │
        │   4. Run subset of beam_ssa_opt passes                   │
        │   5. Inline monomorphic callees (recursive, capped)      │
        │   6. Re-run beam_ssa_pre_codegen + beam_ssa_codegen      │
        │      OR emit new "T2-bytecode" with extra ops            │
        │   7. Hand result back to BeamAsm to JIT into a new blob  │
        └──────────────────────┬───────────────────────────────────┘
                               │
                               ▼
        ┌──────────────────────────────────────────────────────────┐
        │  BeamAsm-as-T2-backend                                   │
        │   - Same emitters, plus a few new ones for trap/guard ops│
        │   - Generates a separate code blob                       │
        │   - Patches Export.addressv to point at T2 code          │
        │   - On deopt: trampoline restores X regs, jumps to       │
        │     BeamAsm code at the recorded BEAM SSA point          │
        └──────────────────────────────────────────────────────────┘
```

The key insight: **T2 emits BEAM SSA (with extensions), reuses
`beam_ssa_pre_codegen` + `beam_ssa_codegen` to lower it back to a
slightly extended BEAM op set, and reuses BeamAsm to emit native
code**. We do *not* write a parallel native code generator. This is
the Sista approach, justified by the Sista paper, validated by the
HiPE post-mortem, and consistent with how ZJIT borrowed YJIT's
backend initially.

## 4. Choices

| Decision | What we choose | Reference / rationale |
|----------|---------------|---------------------|
| Compilation unit | **Method-level** | ZJIT, Maglev, JSC-DFG; Bolz-Tereick says trace JITs aren't worth it given resources. |
| IR style | **CFG-based SSA** | Maglev explicitly chose this over sea-of-nodes for cache friendliness; matches our existing `beam_ssa_*`. |
| IR identity | **BEAM SSA, extended** | HiPE's parallel IR was a maintenance disaster. Sista showed bytecode-to-bytecode works; we already have the SSA. |
| Backend | **Reuse BeamAsm** | Sista reused Cog's backend; ZJIT reused YJIT's. Standard pattern. Custom register allocation is a v2 task. |
| Tier-up trigger | **Per-function entry counter** | JSC's single-counter scheme is simpler than per-call-site and works well. |
| Profile data | **Per-call-site monomorphic-target slot + branch counters** | Borrowed from JSC value profiles + HotSpot MDO. Smallest viable shape. |
| Speculation model | **OSR-style guards (deopt to T1)** | JSC, Maglev, HotSpot all do this. YJIT-style "side-exit-to-interpreter" but the interpreter is BeamAsm. |
| Deopt target | **BeamAsm-compiled code at a known BEAM SSA point** | We don't have an interpreter. BeamAsm is the lowest tier and is correct by construction. |
| Inlining v1 | **Monomorphic only + constant-fun-target inlining + tail-recursion → loop recovery** | ZJIT's v1; Sista's monomorphic case. Constant-fun case is needed for `lists:foldl(F,...)` with literal `F`. |
| Higher-order intrinsics | **AOT-annotated `jit_inline`, initial set = the `sys_core_fold_lists.erl` 10 BIFs** | Reuses existing AOT pattern; evolves to library-author opt-in and AOT auto-inference. |
| Loop optimizations v1 | **Recovery + LICM + unrolling (factor 4-8)** | IV analysis is weak for Erlang's list-walking idioms; unrolling pays off via `test_heap` coalescing and term-creation batching. |
| Polymorphic v2 | **PIC-style switch with N≤4 receivers** | Sista's polymorphic case; JSC's PICs. Roadmap. |
| Recompilation | **Exit-counted with exponential backoff** | JSC's `100 * 2^R` model; HotSpot does the same. |
| Watchpoints | **For module reload, fun reload, BIF rebinding** | JSC's invalidation model. Hooks already exist in `code:purge/1`. |
| Coexistence | **Opt-in, off by default** | ZJIT's deployment model; matches "don't break BeamAsm" goal. |
| Architectures | **aarch64 first, x86_64 later** | aarch64 is where we want it most (Apple Silicon + ARM cloud); x86_64 follows once architecture-agnostic patterns are proven. |
| Code cache | **Separate region from BeamAsm, evictable** | YJIT's bounded budget + GC pattern. |
| Implementation language | **Erlang for optimizer, C++ for runtime hooks** | Sista wrote Scorch in Smalltalk. We have `beam_ssa_*` in Erlang already. The only C++ is the BeamAsm hooks. |

## 5. The IR

We use **`beam_ssa.hrl`'s existing records** as the T2 IR, with three
extensions in `#b_set.anno`:

### 5.1 New `b_set.op` values for T2

| Op | Purpose |
|----|---------|
| `{t2, type_guard, Type}` | Speculate that the operand has type `Type`. On failure, deopt at the current SSA point. |
| `{t2, monomorphic_call, MFA}` | A call site predicted to always reach `{Mod,Fun,Arity}`. Equivalent to a `bif` op for the inlining pass; lowered to a guard + direct call. |
| `{t2, deopt_point, MetaIdx}` | Marks an SSA point where deopt metadata `MetaIdx` lives. Codegen emits no machine code; only used to anchor the metadata table. |

### 5.2 New annotations

| `#b_set.anno` key | Value | Purpose |
|-------------------|-------|---------|
| `t2_deopt` | `{BeamSSAPoint, [{XReg, SSAValue}, ...]}` | At this point, the X-register state should match the listed mapping if we deopt. |
| `t2_inlined_from` | `{Mod, Fun, Arity, BeamSSAPoint}` | This SSA op was inlined from the named function. Used to construct nested deopt frames. |
| `t2_profile_origin` | profile counter ref | For diagnostics / re-profiling. |

`#b_function.anno` extension:

| Key | Value | Purpose |
|-----|-------|---------|
| `jit_inline` | `#{fun_arg_pos => N \| [N,...]}` | This function is a higher-order helper; T2 should inline it when call site has a constant-known fun in argument position(s) `N`. AOT compiler sets this on `lists:foldl/3` etc.; user libraries can add `-jit_inline` attributes. |

### 5.3 Why we don't make a parallel IR

HiPE built Icode + RTL on top of BEAM bytecode. This was a maintenance
disaster; every new BEAM op required updating both Icode and RTL. The
AOT compiler now has an SSA pipeline that does most of what we need.
Reusing BEAM SSA (with the small additions above) means:
- New BEAM ops automatically work in T2 (codegen is via the same path).
- Bug fixes in `beam_ssa_opt` benefit T2 immediately.
- Test infrastructure (`%ssa%` checks, the validator) covers T2 IR.

The Sista paper makes the same argument for SistaV1 ("optimised methods
are still bytecode and load through the same loader") and the launch of
ZJIT's HIR confirms this is now the consensus design for tier-2 JITs of
dynamic languages.

### 5.4 Loop info as analysis (not as IR construct)

T2 does not introduce a `loop` IR node. The IR stays as plain BEAM SSA
blocks with branches; loop structure is recovered by **analysis**, the
same shape as LLVM's `LoopInfo` or Maglev's loop-phis-on-blocks.

A `beam_t2_loops` analysis pass:
- builds the dominator tree,
- finds back-edges (an edge `B → H` where `H` dominates `B`),
- groups blocks by loop header,
- produces a side data structure
  `#{HeaderLabel => #t2_loop{header, latches, exits, body_blocks, parent}}`,
- attaches it to the function via a transient annotation
  (`t2_loop_info` in `b_function.anno`, dropped before lowering).

Subsequent passes (LICM, unrolling, induction-variable analysis if and
when added) consume the loop-info side table. Because the IR underneath
is still flat blocks, **lowering to BEAM ops is a no-op for loop
structure** — the back-edge is still a `b_br`, and BeamAsm emits it
exactly as today.

#### Loop recovery during inlining

When the inliner expands a tail-recursive function (e.g. the recursive
helper inside an inlined `lists:foldl/3`), it should emit a loop *in
place of* nested recursive calls:
- recognise: callee is self-recursive, recursive call is in tail
  position;
- rename the callee's entry block as a loop header inside the caller;
- replace the recursive tail call with a back-jump that updates the
  formal parameters as block-arg-style updates to the header.

Without loop recovery, inlining `lists:foldl/3` produces N levels of
nested recursive expansion until the depth cap fires — much worse than
a single self-jumping loop. With loop recovery, the same input becomes
a flat loop on which LICM and unrolling can fire.

This is the only "structural" thing we add at inlining time. It runs
*as part of* the inliner, not as a separate pass.

## 6. Profiling infrastructure

Changes to BeamAsm. All cheap, all *additive*; if T2 is disabled, none
are emitted.

### 6.1 Per-function entry counter

A single 32-bit counter per function. Incremented on entry. Lives in a
small per-function metadata block (separate from code so it stays in
data cache).

When the counter crosses the tier-up threshold (see §14), the entry
stub branches to the T2 manager via a runtime call.

Implementation: extend the patchable function-prologue region (already
present for tracing) with one more instruction sequence:
```
ldr   tmp, [counter_addr]
add   tmp, tmp, #1
str   tmp, [counter_addr]
cmp   tmp, threshold
b.ge  t2_request_compile
```
~5 instructions, < 1 cache line. Cost: ~1 ns per function entry.
Cheap enough that we can leave it on permanently.

### 6.2 Per-call-site monomorphic-target slot

For each `call_ext`, `call_fun`, and `apply` site, allocate one word
in the function's metadata to hold:
- `0` — never observed.
- `Export*` — observed exactly one target so far.
- `POLY` (special sentinel) — observed two or more distinct targets.

Update on each call: if slot is 0, write the target; if slot equals
target, no-op; otherwise CAS to POLY.

Cost: 1 load + 1 compare + branch. ~2 ns per call. Could be skipped
for small functions or after the function has been T2-compiled.

### 6.3 Branch-frequency counters (optional, v2)

For `is_*`, `select_val`, `select_arity` sites — a 16-bit counter pair
(taken/not-taken). Used by T2's `ssa_opt_dead` analogue to bias branch
ordering and prune dead arms under speculation.

### 6.4 Storage

Each function's profile data lives next to its code in a
`BeamCodeFunction` extension. Lifetime is tied to the module; on
purge, the profile data goes with it.

## 7. The T2 optimizer pipeline

The optimizer is **Erlang code**, runs in the runtime system (not at
compile time), and processes one function at a time.

```
   T2 manager: tier-up request for {M,F,A}
     │
     ▼
   beam_t2_opt:compile({M,F,A}, ProfileData)
     │
     ├─ 1. Fetch BEAM SSA for the function from BEAM file
     │     (or from on-disk cache; SSA is included in BEAM file
     │     since OTP 25 type-information work)
     │
     ├─ 2. Narrow argument types using ProfileData.argTypes
     │     (the call-site monomorphic targets give us actual types,
     │     not 'any')
     │
     ├─ 3. Insert speculation guards at each profile-narrowed type
     │     - emit {t2, type_guard, Type} at each entry
     │     - record X-register snapshot in t2_deopt anno
     │
     ├─ 4. Run a curated subset of beam_ssa_opt passes:
     │       ssa_opt_type_continue   (re-narrow, with new arg types)
     │       ssa_opt_dead             (branch shortcut on better preds)
     │       ssa_opt_cse              (CSE with sharper types)
     │       ssa_opt_record           (more record specialisations)
     │       ssa_opt_alias            (alias analysis)
     │       ssa_opt_destructive_update
     │       ssa_opt_float            (more unboxed-float regions)
     │       ssa_opt_get_tuple_element
     │
     ├─ 5. Inlining pass (NEW; see §9):
     │     - for each {t2, monomorphic_call} site,
     │       fetch callee SSA, alpha-rename, splice in,
     │       wrap with t2_deopt metadata for the inlined frame
     │     - re-run subset of passes 4 inside the new region
     │
     ├─ 6. Lower {t2, type_guard, Type} → existing is_TYPE checks
     │     followed by branches to a deopt block
     │
     ├─ 7. Re-run beam_ssa_pre_codegen + beam_ssa_codegen
     │     to produce BEAM ops + register allocations
     │
     ├─ 8. Build deopt metadata table from t2_deopt annotations
     │     (this is the side-table BeamAsm uses to translate a
     │     T2 deopt to a BeamAsm code address)
     │
     └─ 9. Hand the result to beamasm_t2_codegen() (C++) which
           generates a separate code blob and patches Export.addressv
```

### 7.1 Pipeline in Erlang, not C++

`beam_ssa_opt`, `beam_ssa_type`, `beam_ssa_pre_codegen`, and
`beam_ssa_codegen` are all already in Erlang (`lib/compiler/src/`).
Calling them from a runtime context just means making sure the
compiler app is loaded (which it usually isn't on production node
startup — we'll need a stripped-down version, or we ship the relevant
modules into a runtime app).

This keeps T2 development *much* faster than if it were C++. We can
hot-reload optimizer code, test with simple ETS-driven harnesses,
and reuse all existing test infrastructure. ZJIT has had to write its
optimizer in Rust for the same access patterns; we have a strict
advantage by being a self-hosted language.

### 7.2 Compile thread, not scheduler

T2 compilation runs on a **dirty CPU scheduler** (already part of the
runtime). It's scheduled like a long-running NIF; the calling
process continues executing T1 code while T2 compiles in the
background. When T2 compilation completes, `Export.addressv` is
atomically patched.

If a process is mid-call when its function is being T2'd, that call
finishes in T1 code. The next call to that function picks up the T2
version via the export indirect.

## 8. Speculation, guards, and deopt

### 8.1 Where guards come from

Three sources:
1. **Profile-narrowed arg types** — at function entry, before any
   user code runs.
2. **Inlined-call receivers** — at each inlined call site, guard that
   the receiver matches the predicted target.
3. **Watchpoint-style assumptions** — these emit *no* check; they
   register a dependency. Examples: "module M has not been reloaded
   since compile time", "BIF X has not been replaced".

### 8.2 The deopt metadata format

Side table per T2 blob:
```c
typedef struct {
    void* t2_pc;              // PC in T2 code where this deopt fires
    void* baseline_pc;        // PC in BeamAsm code to resume at
    uint16_t x_reg_count;     // number of X registers to restore
    struct {
        uint16_t x_reg;       // index into the X-reg array
        Eterm value_or_slot;  // either an immediate or a stack slot ref
    } x_regs[];               // packed array
    // for inlined frames: parent frames here, recursively
    uint8_t frame_count;
    struct frame_entry {
        void* baseline_pc;
        uint16_t x_reg_count;
        // ...
    } frames[];
} T2DeoptEntry;
```

On deopt:
1. The T2 code branches to a generic `t2_deopt_dispatch` stub with
   the `T2DeoptEntry*` in a known register.
2. The stub:
   - Restores X registers from `x_regs[]`.
   - If `frame_count > 1`, materialises N nested CP frames on the
     Erlang stack (in reverse, deepest first).
   - Branches to `frames[0].baseline_pc` (the innermost frame, which
     is the one we were "executing" in T2).
3. BeamAsm code resumes execution. The function effectively re-enters
   the unoptimised path.

### 8.3 Inlined-frame deopt — the hard case

If T2 inlined function `g` into `f`, and a guard fires inside the
inlined `g`, deopt has to materialise:
- A CP frame for `f` (so when the resumed `g` returns, we end up back
  in `f`'s baseline PC).
- An X-register state for `g` (so `g` can resume).
- A CP frame for `g` — actually no, if `g` is "currently executing"
  there's no extra CP frame, just the X reg state.

#### Funs: only the speculative case is hard

The Sista FullBlockClosure deopt nightmare arises specifically from
*speculating* on a runtime fun value's identity. If the speculation
lies, deopt has to materialise the original fun, reconstruct the
indirect `call_fun`, and unwind. That's the polymorphic-PIC scenario
in §9 and stays in v2.

The case where a `call_fun` site has a target that is **provably
constant** via SSA constant propagation (e.g. the literal fun argument
to an inlined `lists:foldl/3`) requires no speculation and no extra
deopt machinery — it's ordinary inlining of a known target. This is
covered in v1 (see §9.1).

The audit list for safe constant-fun inlining:
- `erts/emulator/beam/erl_fun.h` and `erl_fun.c` — fun representation.
- `make_fun3` and `call_fun` instructions in `arm/instr_fun.cpp`.
- Closure-capture invariants — when inlining the fun's body, the
  inliner must substitute the captured environment correctly. For a
  fun that captures nothing (the most common case), this is trivial.
  For a fun that captures variables, the captures live as fresh SSA
  values produced at the `make_fun` site; the inliner threads them
  into the inlined body.

What stays out of v1: speculative fun specialisation from runtime
PIC observation. That's the case Sista paid for, and the lesson is to
defer it until the ordinary-inlining path is solid.

### 8.4 Recompilation policy

Each T2 blob has an exit counter. Threshold: `100 * pow(2, R)` where
`R` is the number of times this function has been recompiled (JSC
model).

When threshold is hit:
- The T2 blob is jettisoned (Export.addressv reverts to the BeamAsm
  blob, which never went away).
- Profile counters reset.
- Function continues executing in T1.
- Eventually the entry counter retrips and T2 reconsiders, this time
  with knowledge of *which* assumptions failed (because we record
  the failed-guard kind in the per-function exit-reason buffer).

Loop-triggered exits are weighted more heavily (`5 * pow(2, R)` like
JSC) — exits inside loops are bad and worth avoiding harder.

## 9. Inlining strategy

### 9.1 What we inline in v1

- **Local calls** (intra-module): always inline if size ≤ threshold.
- **Remote calls** to monomorphic targets where the target is a
  pure or near-pure function: inline if size ≤ threshold.
- **`jit_inline`-annotated higher-order helpers** when the relevant
  fun argument is a constant-known fun at the call site. Initial
  annotation set: the 10 BIFs already covered by
  `lib/compiler/src/sys_core_fold_lists.erl` (`lists:all/2`, `any/2`,
  `foreach/2`, `map/2`, `flatmap/2`, `filter/2`, `foldl/3`, `foldr/3`,
  `mapfoldl/3`, `mapfoldr/3`). See §9.5.
- **`call_fun` sites whose target is provably a known fun** via SSA
  constant propagation — e.g. the literal fun threaded into an
  inlined `lists:foldl/3`. No speculation; no extra deopt metadata.
- **Guards** (already inlined by AOT, so this is implicit).

### 9.2 What we don't inline in v1

- **Polymorphic call sites** (PIC-style switch over multiple observed
  receivers). Profile counter is `POLY`. Roadmap, see §17.
- **Speculative fun specialisation** (`call_fun` where the target is
  not statically provable, but observation says it's monomorphic).
  This is the Sista FullBlockClosure case; v2.
- **`apply/3` with non-constant arguments.** Can't predict target.
- **Naive recursive expansion.** A self-recursive callee should be
  inlined via the loop-recovery path (§5.4), not expanded N levels
  deep. Non-tail recursion stays out of v1.
- **Calls into non-purgable modules' BIFs that we haven't audited.**

### 9.3 Heuristics

- Size threshold: SSA `b_set` count ≤ 24 (matches AOT inliner default).
- Depth cap: 3 levels.
- Total expansion cap: 5× original function's SSA size.
- *Skip inlining* if the callee has been observed to deopt in a
  previous T2 compile (cheap death-cycle prevention).
- **Tail-recursive callees → loop recovery** (§5.4), not nested
  expansion.

### 9.4 What it unlocks

The single biggest win from inlining is **constant folding across what
were function boundaries**. Consider:
```erlang
double(X) -> X * 2.
quadruple(X) -> double(double(X)).
```
After inlining, `quadruple/1` becomes `X * 2 * 2`. After type
narrowing (`X` is `#t_integer{elements={1, 100}}` from profile),
`ssa_opt_dead` and `beam_ssa_type` together fold this to `X * 4` and
the small-integer fast path is a single `mul` with no overflow check.

The same pattern collapses: accessor chains, small wrapper functions,
guard helpers, builder functions, options-handling helpers. Empirically
(see HiPErJiT's evaluation, see ZJIT's launch numbers) this is where
realistic Erlang code wins.

### 9.5 Higher-order intrinsics: from hardcoded list to annotation

The AOT compiler already inlines a fixed set of `lists` higher-order
BIFs when the user opts in via `-compile(inline_list_funcs)`. The
implementation is `lib/compiler/src/sys_core_fold_lists.erl` (~400
lines of hand-written Core Erlang expansions). T2 reuses this pattern
but evolves it in two ways:

1. **Always-on for hot code.** T2 ignores the `inline_list_funcs`
   compile flag — if a function is hot enough to T2-compile and a
   `jit_inline`-annotated higher-order helper appears with a
   constant-known fun argument, the inlining happens. This makes the
   typical Erlang idiom (`lists:foldl(fun(...) -> ... end, ...)`)
   fast without source changes.

2. **Annotation-driven, not hardcoded.** The AOT compiler is updated
   to emit `jit_inline => #{fun_arg_pos => N}` on `b_function.anno`
   for every function it knows is a higher-order helper. Initial set
   matches `sys_core_fold_lists.erl`; this is the v1 shipping list.
   Library authors can opt their own helpers in via a `-jit_inline`
   module attribute. Future work auto-infers the annotation from the
   "function calls `call_fun` on a parameter" pattern.

Implementation reuse: the SSA expansion shapes for the lists BIFs
should be derived from `sys_core_fold_lists.erl` directly (translate
the Core Erlang expansions to BEAM SSA equivalents, or generate them
via the existing `core_to_ssa` pipeline at JIT-init time and cache).

Important benchmarking note: when measuring T2 wins on
higher-order-heavy code, **disable** `-compile(inline_list_funcs)` in
the baseline so we don't conflate "T2 is great" with "the AOT had
this turned off".

### 9.6 Loop optimizations (v1 set)

Once loop recovery (§5.4) has produced a flat loop, T2 runs:

#### LICM (loop-invariant code motion)

For each block in a loop body, hoist instructions whose operands are
all defined outside the loop or are themselves hoisted invariants.
Cheap (single forward walk over the loop body once dominators are in
hand) and pays for itself by hoisting:
- type guards on parameters that don't change per iteration,
- captured-environment loads from inlined funs,
- constant subexpressions the AOT couldn't see before inlining.

#### Loop unrolling

This is the **first-class loop optimization for Erlang**, not LICM,
because Erlang's per-iteration costs are dominated by amortisable
work, not addressing computation:

1. **GC test coalescing.** `test_heap` is emitted per allocation site.
   Without unrolling, each iteration's allocations get their own heap
   tests. Unrolling K iterations into one body lets us emit a single
   `test_heap(K * iter_alloc_words)` covering the whole batch — a
   substantial win for any loop that allocates per element
   (`lists:map/2`, `lists:filter/2` keeping, list comprehensions,
   binary construction).
2. **Spine traversal batching.** `[H1|T1] = L; [H2|T2] = T1; ...`
   collapses into fewer comparisons, fewer back-edges, and on aarch64
   lets the load-store unit pipeline the cell loads.
3. **Term creation batching.** Building K cons cells in one body
   lets us advance HTOP once (`add htop, htop, K*16`) instead of K
   times.
4. **Branch overhead amortisation.** One taken back-edge per K
   elements instead of one per element.

Heuristics:
- Default unroll factor: 4.
- Increase up to 8 for very small bodies (e.g. `foldl` with `+`).
- Skip entirely for bodies containing calls (other than known-pure
  inlined ones), variable-cost work, or anything that already exits
  the loop early.
- After unrolling, **re-run heap-test placement** so allocations
  within the unrolled body are covered by a single combined
  `test_heap` at the unrolled-loop entry. This is the win that
  justifies most of the unrolling work.
- Watch the code-cache budget: factor-4 unrolling roughly quadruples
  body size, and the T2 cache is bounded.

#### Induction variable analysis — *not* in v1

In imperative-language JITs, IV analysis is high-value because of
array indexing patterns (`for(i=0; i<n; i++) arr[i]`) where strength
reduction turns `arr[i]` into `*p++`. Erlang doesn't iterate flat
arrays — it walks lists by pointer-chasing cons cells, with the list
itself as the "loop variable". There's no addressing computation to
strength-reduce.

The places IV analysis would help are real but a long tail:
- binary matching position tracking,
- the rare integer-counter loop,
- map iteration if/when we add specialised iterator support.

It moves to v2.

## 10. Code generation strategy

### 10.1 v1 — bytecode-to-bytecode + reuse BeamAsm

T2 emits BEAM SSA with extensions (§5), runs `beam_ssa_pre_codegen`
and `beam_ssa_codegen` to lower to BEAM ops, hands the result to
BeamAsm. BeamAsm needs minor extensions:

- New emitters for `{t2, type_guard, Type}` → existing is-type-test +
  branch-to-deopt-stub.
- New emitter for `{t2, deopt_point, ...}` → no code, just record the
  PC + metadata index.
- A shared `t2_deopt_dispatch` global fragment.
- A separate code allocator region (so T2 code can be evicted
  independently).

Estimated LOC:
- 5 new emitter methods × (~50 lines each ARM, ~50 lines x86_64) =
  ~500 lines of C++.
- The `t2_deopt_dispatch` fragment: ~200 lines of C++.

Total BeamAsm extension: < 1 KLOC.

### 10.2 v2 — possible native code generator

Once v1 is shipping and we've measured what the bytecode-to-bytecode
approach leaves on the table, v2 may add a direct SSA-to-native
codegen for hot regions (Maglev's approach). Expected wins:
- Cross-instruction register allocation (don't flush X regs at every
  C call boundary).
- Untagged-value tracking through optimization regions.
- Specialised arithmetic instructions (e.g. ARM64's `madd` for FMA).

This is **roadmap, not v1**. v1's bytecode-to-bytecode approach is
strictly easier to ship and gives us the foundation to measure.

### 10.3 Why not LLVM / Cranelift / a separate backend

- LLVM: licensing OK now (Apache 2.0 with exception), but compile-time
  cost is way too high for a JIT, and the dependency footprint is
  enormous.
- Cranelift: workable, but still adds a big dependency and we don't
  benefit from its multi-language design. asmjit + reuse is smaller.
- Separate backend: HiPE lesson. Don't.

## 11. Calling convention and frame layout

### 11.1 Same as BeamAsm

This is an architectural commitment. T2 code uses:
- `x19` = scheduler_registers
- `x20` = E (Erlang stack pointer)
- `x21` = c_p
- `x22` = FCALLS
- `x23` = HTOP
- `x25–x28` = XREG0..XREG3
- `x15–x17` = XREG4..XREG5

A T2 function is **callable by T1 code as if it were T1 code** —
because it has the same calling convention.

### 11.2 What changes inside a T2 function

Within a T2-optimised region, T2 may:
- Keep additional X-register values in callee-save registers across
  C-call boundaries (saving/restoring around the call) — *if* the
  optimization region is large enough to amortise.
- Use untagged values in temporaries (not in X registers).
- Defer X-register flushes that BeamAsm would do eagerly.

But at every **deopt point**, the X register state must be
*reconstructable* as if T1 had executed up to the corresponding BEAM
SSA point. This is what the `t2_deopt` annotation tracks.

### 11.3 GC

GC walks the X-register array (via `c_p->scheduler_registers->x_reg_array`)
and the Erlang stack from `c_p->stop` to `c_p->stack_end`. Both are
already understood by BeamAsm. T2 code:
- Must flush X registers before any GC point (allocation, function
  call that may GC).
- Must keep untagged values out of X registers (they'd confuse the
  GC scanner).
- Must keep Y registers tagged (they live on the Erlang stack and
  will be scanned).

Conservatively, T2 v1 keeps the same GC discipline as BeamAsm. We don't
attempt to teach GC about untagged spill slots in v1.

## 12. Code cache and lifecycle

### 12.1 Memory budget

- Default: 64 MB for T2 code (configurable via emulator flag
  `+JT2cache <n>` and runtime `system_flag/2`).
- Allocator: separate `JitAllocator` instance from BeamAsm's, with
  its own RX/RW dual-mapping pool.
- Block size: 32 MB (matches BeamAsm).

### 12.2 Eviction

When the budget is full and a new T2 compile is requested:
1. Find the T2 blob with the lowest "useful work / bytes" score
   (recent execution count / blob size).
2. Revert its export entry to the T1 BeamAsm blob.
3. Free its memory after a grace period (in case another scheduler
   was mid-execution — same wait-for-quiescence pattern as code
   purge).
4. Compile the new function.

### 12.3 Per-function metadata

Each T2 blob has:
- The deopt metadata table (§8.2).
- The execution count (for re-evaluating "is this still hot?").
- The exit count (for jettison).
- The watchpoint list (modules / funs / BIFs we depend on).
- Backpointer to the T1 blob (for fast revert).

## 13. Module reload and code purge

This is where HiPE lost. Erlang permits `code:purge/1` and atomic
reload. Any T2 optimization that depended on assumptions about *some
other module* must be invalidated when that module is reloaded.

### 13.1 Watchpoint registration

When T2 compiles a function and folds in (e.g.) a constant from
another module, an inlined call to another module's function, or a
BIF whose binding could change — it registers a watchpoint:

```
watchpoints[t2_blob] = {
    [{module_loaded, ModName} -> invalidate]
    [{module_loaded, ModName, FunName, Arity} -> invalidate]  // finer-grained
    [{bif_rebound, ModName, FunName, Arity} -> invalidate]
}
```

### 13.2 Invalidation hook

In `code_server.erl` / `beam_load.c` / wherever module load happens:
after the new module is installed but before it's exposed, walk the
T2 watchpoint table and jettison any T2 blobs that watched the old
module.

For affected functions:
1. Revert `Export.addressv` to the T1 blob.
2. Grace-period before freeing the T2 blob.
3. Reset the function's tier-up counter so it can be re-optimized
   later with the new module's behaviour.

### 13.3 Granularity

Per-function watchpoints, not per-module-pair. The cost of a
mass-invalidation when (e.g.) `lists` is reloaded is acceptable; in
practice this is rare.

### 13.4 Hot code upgrade

Hot code upgrade in OTP releases happens via `release_handler` and
ultimately calls `code:purge/1` and `code:load_file/1`. The
invalidation hook above covers it — T2 blobs depending on the
upgraded module die, and re-tier later under the new module.

## 14. Tier-up triggers and heuristics

### 14.1 Threshold

Per-function entry counter, threshold scaled by:
```
threshold = base * sqrt(size + 1) * pow(2, recompile_count)
            * (M / (M - U))
```
- `base` = 1000 (tunable; matches JSC's Baseline→DFG threshold).
- `size` = number of `b_set` ops in the function's SSA.
- `recompile_count` = number of prior T2 compiles for this function.
- `M`, `U` = total / used bytes of T2 code cache.

### 14.2 Profile saturation

Before tier-up, check that profile data is "saturated":
- ≥ 75 % of call sites have observations.
- Function has been running long enough that its argument-type bucket
  has converged.

If not saturated, reset counters, give it more time (max 5 retries —
JSC's pattern).

### 14.3 Compile queue

T2 compiles are queued; one runs at a time per dirty CPU scheduler.
If the queue is full, further requests get dropped (function continues
in T1; counter trips again later).

### 14.4 OSR (deferred)

OSR-entry to T2 mid-loop is *not* in v1. Hot loops that don't span a
tier-up-triggering function call (e.g. `fold` where the fold body is
inlined into the caller — but the caller isn't called often) won't
get T2'd in v1. This is acceptable given Erlang's idiom of expressing
loops as recursive function calls.

If we find this is a real issue (some workloads with explicit loops
inside hot functions), OSR is a v2 task.

## 15. Observability

We must build observability from day one. YJIT's experience with
`runtime_stats` is the model. New ETS table `erlang:t2_stats/0`:

```erlang
#{
  t2_blobs_compiled => N,
  t2_blobs_active => N,
  t2_blobs_jettisoned => N,
  t2_compile_time_us => Total,
  t2_compile_failures => N,
  t2_code_cache_size => Bytes,
  t2_code_cache_used => Bytes,
  t2_code_gc_count => N,
  t2_deopts_total => N,
  t2_deopts_by_reason => #{ TypeGuardFail => N, Watchpoint => N, ... },
  ratio_in_t2 => Float % time spent in T2 code
}
```

Plus per-function:
- `erlang:t2_info/3 :: (M,F,A) -> tier1 | {tier2, Stats}`
  Returns counters, deopt history, current code address.

And debugging hooks:
- `+JT2trace_compiles` — log every compile.
- `+JT2trace_deopts` — log every deopt with reason.
- `+JT2dump_ssa <Mod:Fun/Arity>` — dump the optimized SSA for a
  specific function.

The compile-side IR can be dumped via the same `%ssa%` infrastructure
the AOT compiler uses for optimization-test verification.

## 16. Implementation phases

Each phase has a "ships when" criterion. Phases are sequential.

### Phase 0 — Plumbing (≈4 weeks)
Goal: nothing observable, but all infrastructure in place.
- Add per-function counter slot to BeamAsm function metadata.
- Add per-call-site monomorphic-target slot.
- Wire emitter changes for ARM64 (counter increment, target observation).
- Add T2 manager Erlang process (does nothing yet).
- Add `+JT2enable` flag (does nothing yet).
- All existing tests continue to pass with `+JT2enable false`.

### Phase 1 — Identity transform (≈4 weeks)
Goal: T2 compiles a function but produces equivalent code.
- T2 manager listens for compile requests.
- T2 optimizer fetches BEAM SSA, runs no passes, runs codegen,
  hands result to BeamAsm via new `beamasm_compile_t2_blob()` C API.
- Patches `Export.addressv` to point at the new blob.
- Verify: hot functions get re-compiled into T2; output is bit-for-bit
  identical (or nearly so) to T1 output.
- This proves: profile collection works, compile pipeline works,
  blob installation works, eviction works.

### Phase 2 — Type narrowing (≈4 weeks)
Goal: T2-compiled functions are *faster* than T1.
- T2 narrows arg types from profile data.
- T2 emits `{t2, type_guard, Type}` at function entry.
- T2 lowers guards to existing is-type tests.
- T2 deopt: simplest possible — fall through to T1 entry point with
  X registers reset to entry state.
- Run `beam_ssa_type:opt_continue` + `beam_ssa_dead` on the
  narrowed-type SSA.
- Benchmark: should see ~10-20% speedup on type-narrowable hot paths
  (small int arithmetic, list traversal). If not, the framework is
  wrong; iterate.

### Phase 3 — Inlining MVP (≈8 weeks)
Goal: monomorphic call inlining + higher-order intrinsics work
end-to-end.
- New SSA pass: `beam_t2_inline.erl`.
- For each `{t2, monomorphic_call, MFA}` site, inline the callee.
- Build per-frame deopt metadata.
- Implement multi-frame deopt dispatch (the hard part).
- Restrict to `call_ext` to known M:F:A; no *speculative* fun
  inlining yet.
- AOT compiler change: emit `jit_inline` annotation on the existing
  `sys_core_fold_lists.erl` set of higher-order BIFs.
- T2 inliner recognises `jit_inline`-annotated callees with a
  constant-known fun argument and inlines them.
- After such an inline, recognise tail recursion and emit a loop via
  loop recovery (§5.4); follow with constant-fun propagation that
  collapses the inlined `call_fun` into a direct call to the fun's
  body.
- Build the `beam_t2_loops` analysis pass.
- Benchmark: big wins on accessor-heavy code, on `lists:foldl`/
  `lists:map`-heavy code, and on gen_server callback dispatch.

### Phase 3.5 — Loop optimizations (≈4 weeks)
Goal: loop unrolling + LICM on T2-recovered loops.
- LICM pass consuming `beam_t2_loops` info.
- Loop unrolling pass with `test_heap` re-coalescing afterwards.
- Heuristics for unroll factor (default 4, up to 8 for tiny bodies,
  skip for impure bodies).
- Benchmark: `lists:map`/comprehension-heavy code with allocation in
  the body should show measurable wins from `test_heap` coalescing
  alone.

### Phase 4 — Production polish (≈8 weeks)
Goal: good enough to enable for testing.
- Code cache eviction.
- Watchpoints + invalidation on module reload.
- Recompilation backoff.
- Observability infrastructure (`erlang:t2_stats/0`, etc.).
- Documentation.
- Integration testing with realistic workloads (RabbitMQ, Cowboy
  HTTP server, OTP test suite).

### Phase 5 — Polymorphic specialization (v2)
Goal: PIC-style specialisation at polymorphic sites.
- Extended profile slot: small array of N observed targets.
- Inlining splits the call site into a switch over target with one
  inline copy per branch.
- Roadmap, not v1.

### Phase 6 — Native codegen (v2/v3)
Goal: bypass `beam_ssa_codegen` for hot regions, emit machine code
directly with cross-instruction register allocation.
- Roadmap, not v1.

### Phase 7 — x86_64 (v2)
Goal: same T2 semantics on x86_64.
- Most work is in BeamAsm extensions for the new emitters.
- Calling convention is already established for x86_64 BeamAsm.
- Test on Linux x86_64.

### Sequencing rationale

The phases are designed so that **each phase ships something
measurable and useful**. Phase 1 ships an identity transform — a
massive sanity check that proves the plumbing is right before any
optimization touches it. Phase 2 ships the smallest useful
optimization (type narrowing) without inlining complexity. Phase 3
adds the big-win optimization (inlining) once the rest is stable.
Phase 4 makes it production-ready.

If at any phase we find the design is wrong, we cut early. Phases 0-2
together (~12 weeks) should validate or invalidate the
bytecode-to-bytecode-via-BeamAsm approach. Phases 3 + 3.5 (~12 weeks)
together prove the inlining + loop-recovery + unrolling pipeline,
which is where the bulk of the realistic perf wins live.

## 17. Risks and open questions

### High-impact risks

1. **Deopt metadata size explodes.** Inlined frames add per-call
   metadata; if the average T2 blob's metadata is comparable to its
   code size, we burn the code cache. **Mitigation**: measure early
   in Phase 3; cap inline depth aggressively if metadata ratio gets
   bad; design metadata to be lazily decoded (compressed varints).
2. **Compile-time overhead is too high.** If T2 takes 100 ms per
   function, the dirty CPU scheduler is saturated under load.
   **Mitigation**: measure in Phase 1; cap compile time per blob
   (abort after N ms); reduce optimizer pass count on overrun.
3. **Deopt is too frequent.** Phase changes in production make T2
   constantly bounce. **Mitigation**: strict recompile backoff;
   per-blob exit-rate telemetry; "demote to T1" if exit rate >
   threshold.
4. **Fun deopt is intractable** (the Sista FullBlockClosure issue
   in our codebase). **Mitigation**: audit fun representation in
   Phase 0; if hard, defer fun inlining to v2; document constraints.
5. **`beam_ssa_*` pass invariants broken by T2 extensions.** Adding
   new ops or annotations may surprise other passes. **Mitigation**:
   add new ops as a separate `t2_*` op tag space; run validator
   between passes; comprehensive test suite.

### Open questions to resolve before Phase 1

- Where does the T2 manager process live in the supervision tree?
  (probably under `kernel`'s code app)
- Where does T2's stripped-down compiler app live? Can we just call
  the existing `lib/compiler/src/beam_ssa_*` modules directly, or do
  we need a "compiler-runtime" subset to avoid pulling in the
  parser, lexer, etc.?
- Is there an existing C API for "give me the BEAM SSA of this
  function from a loaded BEAM file", or do we need to add one? (The
  AOT compiler embeds optional SSA in BEAM files via
  `+type_information`; we need to confirm SSA itself is preserved.)
- Counter slot placement — inside the function prologue patch area
  (limited bytes) or in a side table indexed by code address? Side
  table is more flexible but adds a load.
- What's the right `+type_information` setting to require? T2 wants
  rich SSA; can we mandate this for any module that wants T2?

### Open questions for later phases

- OSR-entry policy: when (if ever) is it worth implementing?
- BIF inlining: which BIFs are pure enough to inline as HIR ops
  (Maglev/ZJIT's DOMJIT-style approach)? Plausible candidates:
  `length/1`, `tuple_size/1`, `element/2`, `is_*/1`, basic
  arithmetic.
- ETS / message-passing optimization: out of v1 scope but is the
  long-tail win for real Erlang.
- Cross-process speculation: probably never; processes have
  independent heaps and isolated state.

## 18. Out of scope

Explicit non-goals so we don't scope-creep:

- A T3 / FTL-equivalent. Maybe one day; not now.
- LLVM dependency. Hard no.
- A separate IR family. Reuse BEAM SSA.
- Cross-module deep optimizations beyond what watchpoint
  invalidation can revert.
- Removal or significant refactor of BeamAsm. T1 stays.
- Speedups in message-passing, ETS, GC, NIFs. T2 is for SSA-
  optimisable code; runtime BIFs are someone else's project.
- Speedups for cold code (init, scripts, one-shot batch jobs). T2
  serves long-running hot code.
- A new bytecode format. We extend BEAM SSA, not BEAM bytecode.
- An interpreter on aarch64. BeamAsm is the lowest tier.
