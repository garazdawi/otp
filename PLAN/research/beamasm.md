# BeamAsm — The Current Erlang JIT (the "T1")

Sources (all in this repo unless noted):
- `erts/emulator/beam/jit/beam_asm.h` — JIT C entry-point declarations.
- `erts/emulator/beam/jit/beam_jit_common.{hpp,cpp}` — shared assembler base class.
- `erts/emulator/beam/jit/arm/beam_asm.hpp` — ARM64 register assignments and ABI.
- `erts/emulator/beam/jit/arm/instr_*.cpp` — per-instruction emitters.
- `erts/emulator/beam/jit/beam_asm_module.cpp` — the per-module assembler.
- `erts/emulator/beam/jit/beam_jit_main.cpp` — global init / allocator setup.
- `erts/emulator/beam/jit/beam_jit_metadata.cpp` — perf/gdb integration.
- ["The Road to the JIT" — erlang.org blog](https://www.erlang.org/blog/the-road-to-the-jit/).

This is the reference for what is *already* in the system. T2 sits on top
of this and reuses much of it, so understanding what BeamAsm provides (and
deliberately does *not* provide) is critical.

## 1. Style: template JIT with no IR

BeamAsm is a **non-optimising, template-style** JIT built on **asmjit**.
Every BEAM instruction has a hand-written C++ emitter (`emit_xxx`) that
emits a fixed pattern of native instructions. There is no instruction
fusion, no peephole optimization across BEAM ops, no data-flow analysis,
no register allocation beyond the fixed assignments below.

Per the team's design philosophy ("The Road to the JIT"):
- Full coverage: every loaded module is JIT-compiled at load time. There
  is no interpreter fallback in mode-mixed mode; BeamAsm is *the*
  execution engine.
- A simple, obvious mapping from BEAM ops → machine code is more
  important than peak machine-code quality.
- The BEAM bytecode format itself (and the AOT compiler's type info) is
  the optimization surface; the JIT just lowers.

This is a lot like JSC's *Baseline* tier: per-bytecode templates, no
optimization, the canonical reference for "what running BEAM means".

## 3. Compilation flow

1. Loader calls `beamasm_new_assembler(mod, num_labels, num_functions, beam_file)`.
   Allocates a `BeamModuleAssembler` holding an asmjit `CodeHolder`,
   pre-creates labels for every BEAM label, sets up `.text`/`.rodata`.
2. For each BEAM instruction, loader calls `beamasm_emit(ba, opcode, op)`.
   The dispatch is a giant switch generated by build-time tooling
   (`beamasm_emit.h`) — one case per specific BEAM opcode, calling the
   matching `emit_xxx` method.
3. After all instructions are emitted: `beamasm_codegen(...)`. This:
   - Flattens asmjit sections, resolves cross-section fixups.
   - Allocates RX (executable) + RW (writable) memory from a global
     `JitAllocator` — typically dual-mapping under W^X enforcement.
   - Relocates and copies the code into the RX/RW pair.
4. Loader calls `beamasm_patch_import` / `beamasm_patch_literal` /
   `beamasm_patch_lambda` to fill in references that depend on other
   modules / literals / funs.
5. Memory is sealed; metadata is registered (`beamasm_register_metadata`)
   for perf and gdb.

Allocator: `JitAllocator` from asmjit, configured via `pick_allocator()`
in `beam_jit_main.cpp`. 32 MB blocks, dual or single mapping depending
on platform.

## 4. ARM64 calling convention

From `arm/beam_asm.hpp`. Callee-saved registers are reserved for runtime
state across the entire JIT-compiled program:

| Reg | Role |
|-----|------|
| `x19` | `scheduler_registers` (base of X-register array) |
| `x20` | `E` — Erlang stack pointer (grows down) |
| `x21` | `c_p` — current Process* |
| `x22` | `FCALLS` — reductions counter |
| `x23` | `HTOP` — heap top |
| `x24` | `active_code_ix` — code index (used for tracing redirect) |
| `x25–x28` | `XREG0..XREG3` — first 4 X-registers in CPU regs (DEBUG: x25-x27 + x15) |
| `x15–x17` | `XREG4..XREG5` (caller-save X regs, flushed before C calls) |
| `x0–x7` | `ARG1..ARG8` — C-call argument regs |
| `x8–x13` | `TMP1..TMP6` — temporaries |
| `x14` | `SUPER_TMP` — destroyed by helpers |
| `x30` | LR — return address |
| `d8–d15` | `FREG0..FREG7` — first 8 F-registers in FPU regs |

X registers beyond #5 spill to a flat array at `x19 + offsetof(SchedulerRegisters, x_reg_array)`.
F registers beyond #7 likewise.

Erlang stack ("Y registers", CP frames) lives at `x20` and grows
downward. CP (continuation pointer) is a tagged `Eterm` value pushed on
the Erlang stack; it is *not* a CPU register.

`emit_enter_runtime()` / `emit_leave_runtime()` flush/restore X-regs to
the array around any C call; they also save/restore HTOP and FCALLS as
needed.

## 5. Per-instruction emitters: examples

Direct call (same module):
```cpp
void BeamModuleAssembler::emit_i_call(const ArgLabel &CallTarget) {
    erlang_call(resolve_beam_label(CallTarget, disp128MB));
}
```
Just a `bl` to a resolved label, with veneer support if it doesn't fit
in 128 MB.

Remote call (export table):
```cpp
void BeamModuleAssembler::emit_i_call_ext(const ArgExport &Exp) {
    mov_arg(ARG1, Exp);
    a64::Mem target = emit_setup_dispatchable_call(ARG1);
    erlang_call(target);
}
```
Dereferences `Export->addressv[active_code_ix]` — i.e. the indirect
dispatch *also* picks the right code version under hot-loading. This is
a big architectural feature: any code transition (interpret → JIT →
trace → other-version) is mediated through the export table.

Tail call:
```cpp
void BeamModuleAssembler::emit_i_call_only(const ArgLabel &CallTarget) {
    emit_leave_erlang_frame();
    a.b(resolve_beam_label(CallTarget, disp128MB));
    mark_unreachable();
}
```

Reduction yield (`i_test_yield`):
```cpp
void BeamModuleAssembler::emit_i_test_yield() {
    a.adr(ARG3, current_label);
    a.subs(FCALLS, FCALLS, imm(1));
    a.b_le(resolve_fragment(ga->get_i_test_yield_shared(), disp1MB));
}
```
Decrement, branch-if-≤0 to a shared fragment that saves the current PC
to `c_p->i` and switches out.

## 6. Calls, BIFs, NIFs

- **Direct intra-module call**: relative `bl`, ~128 MB veneer table.
- **Inter-module call**: `Export->addressv[active_code_ix]` indirect.
- **BIF guard call** (no GC, no reduction): `emit_enter_runtime` →
  `dynamic_runtime_call<3>` → `emit_leave_runtime`. Returns
  `THE_NON_VALUE` on failure; checked by branch.
- **BIF body call**: same shape but with reductions update and
  exception lookup.
- **Apply / dynamic dispatch**: trampolines through `apply()` C
  function.
- **NIF**: function prologue is padded to `BEAM_NATIVE_MIN_FUNC_SZ`
  words so a NIF stub can be patched in-place when the NIF is loaded.

## 7. Exception handling

Catches use `c_p->catches` counter + Y-register-stored handler label.
`emit_raise_exception()` jumps to a global fragment that calls
`handle_error()`, which sets `c_p->freason/fvalue/ftrace`, walks the
Erlang stack for catch frames, and either returns a handler PC (jump
to it) or schedules out.

The exception path is *not* per-emitter: there's a single shared
"raise_exception" fragment in `arm/beam_asm_global.cpp` that all
emitters branch to.

## 8. Existing optimization surfaces

Even though there's no IR, BeamAsm specialises locally based on what
the loader hands it:

- `getTypeUnion()`, `always_small()`, `always_immediate()` (from
  `beam_jit_common.hpp`) read the AOT compiler's per-operand type
  information (carried in the BEAM file in OTP 25+) and let emitters
  pick fast paths. Examples:
  - `+` between two known-small operands skips the overflow check on
    the small-fastpath, branches to a slow path on overflow.
  - `is_integer/1` guard is omitted entirely if the type already
    proves integer-ness.
- Operand encoding picks small immediates / literal pool / `mov_imm`
  appropriately.
- asmjit's `EncodingOptions::kOptimizeForSize` chooses short branch
  encodings.
- ARM-specific operand validity (bitfield/logical immediate forms,
  `csel`/`ccmp` for short conditional sequences).

What BeamAsm does *not* do:
- No CSE across BEAM ops.
- No DCE across BEAM ops.
- No register allocation (X-reg → CPU-reg mapping is fixed).
- No inlining of any kind.
- No constant propagation across instructions.
- No specialised arithmetic beyond per-instruction overflow-fast-path.

These are exactly the spaces a T2 fills.

## 9. Tracing/profiling support already present

- **perf integration**: `beam_jit_metadata.cpp` writes perf.map files
  on Linux, registers JIT entries via `__jit_debug_register_code`
  (gdb), supports frame pointers via `+JPfp`.
- **Line numbers**: `emit_line()` records source-line ↔ code-offset
  for stack traces.
- **Tracing**: function prologues have a 3-instruction patchable
  region (`{short jump, nop, call breakpoint_handler}`) used to
  enable per-function trace breakpoints without invalidating code.
  Same prologue can be used for any other "before this function
  runs, do X" hook.
- **Call tracing**: `instr_trace.cpp` has emitters for `call_trace`,
  `return_trace`, `return_to_trace`.
- **Save-calls** in `Export->addressv` redirects to a fragment that
  records the call into the call buffer — used by `process_info(P,
  last_calls)`.

For T2 we can piggy-back on:
- The export table indirection (free hot-swap of generated code).
- The patchable function prologue (can install a tier-up counter or
  a redirect).
- The metadata registration (gdb/perf gets T2 code for free).

## 10. What signals must be added for T2

Not present today; need to be added:

1. **Per-function execution counter.** Increment on entry, threshold
   triggers T2 compile. Could live in the patchable prologue area.
2. **Per-call-site type observation.** Most simply: at `call_ext` /
   `call_fun` / `apply`, record the receiver class / fun target into
   a small profile slot in the function's metadata.
3. **Branch frequency counters** at `is_*` and `select_val`/
   `select_arity` sites — for biased branch ordering and dead-arm
   removal under speculation.
4. **OSR exit handler / deopt entry point.** A way for T2 code to
   transition mid-function back into BeamAsm code. Requires:
   - A canonical deopt-frame format (X-reg snapshot + BEAM SSA point
     descriptor).
   - A trampoline that takes the snapshot and jumps to the BeamAsm
     code at the right BEAM label.
5. **Module-level T2 blob installation.** A second `addressv[]` slot
   per export, or per-function dispatch table, that lets the export
   table point at T2 code if installed and fall back to BeamAsm
   otherwise.
6. **Code purge integration.** When a module is purged or reloaded,
   any T2 blobs that depend on it must be invalidated.

All of these are *additions* — none requires rewriting the existing
BeamAsm path. The existing path stays the source of truth; T2 is an
optional accelerator.

## 11. Memory and code cache

`JitAllocator` blocks of 32 MB, dual-mapped on most platforms. Code is
not currently moved/compacted; modules unloaded → blocks freed.

For T2 we need to add a separate allocator (or a separate region in
the same allocator) so T2 blobs can be evicted independently and we
can have a different memory budget for them. We probably want T2 code
to be evictable under pressure — fall back to BeamAsm when evicted.

## 12. Summary for T2 design

BeamAsm is small, clean, and deliberately incomplete. The pieces T2
needs to add are:
- An IR layer above BEAM SSA (or reusing it directly).
- An optimizer (the tier-2 passes).
- A per-architecture code generator from the optimised IR — *or* a
  bytecode-to-bytecode strategy that emits something BeamAsm can
  still ingest (the Sista approach).
- Profiling counters at well-defined points in BeamAsm.
- A deopt mechanism that lands back in BeamAsm.
- A code-cache management policy for T2 blobs.

Crucially, *none of these break what BeamAsm does today*. T2 is
purely additive.
