# V8 — Sparkplug, Maglev, TurboFan

Sources:
- [Maglev — V8's Fastest Optimizing JIT (2023)](https://v8.dev/blog/maglev) (primary)
- [Digging into the TurboFan JIT](https://v8.dev/blog/turbofan-jit)
- [V8 Sparkplug intro (InfoQ, 2021)](https://www.infoq.com/news/2021/06/v8-sparkplug-compiler/)
- [Holiday season 2023 retrospective](https://v8.dev/blog/holiday-season-2023)

V8 has the most-tiers-in-production of any dynamic-language VM:
**Ignition** (interpreter) → **Sparkplug** (baseline JIT) → **Maglev**
(fast optimizer) → **TurboFan** (peak optimizer). Maglev is the most
relevant tier for our T2 design because it occupies *exactly* the niche
we want for an Erlang T2: a fast SSA-based optimizer that consumes
profile data, sits above a baseline JIT, and bails to a lower tier on
deopt.

## 1. Tier hierarchy

| Tier | What | Compile cost | Runtime perf (JetStream baseline 1.0) |
|------|------|--------------|----------------------------------------|
| Ignition | bytecode interpreter | n/a | 1.0 |
| Sparkplug | baseline template JIT (no IR) | ~1× | 1.45 |
| Maglev | CFG-SSA optimizer with type feedback | ~10× Sparkplug | between Sparkplug and TurboFan |
| TurboFan | sea-of-nodes optimizer with all of B3-style passes | ~10× Maglev (~100× Sparkplug) | 4.35 |

So Maglev is "10× slower to compile than Sparkplug, 10× faster to
compile than TurboFan, and produces code somewhere between the two on
runtime perf". The blog claims **Speedometer** (a more realistic web
benchmark) shows the Maglev tier closes a substantial fraction of the
Sparkplug→TurboFan gap that the simpler benchmarks hide.

## 2. Why Maglev was added

The official rationale:
> "Sparkplug's simplicity imposes a relatively low upper limit on the
> speedup it can provide. … TurboFan compiles ~10x slower than
> Sparkplug, creating a deployment cost. Functions may not run long
> enough to justify TurboFan compilation before the page becomes
> interactive."

In our terms: the existing fast tier (Sparkplug ≡ BeamAsm) tops out at a
modest speedup; the heavyweight optimizer (TurboFan ≡ a full LLVM-style
backend) is too expensive to deploy widely. The middle tier closes that
gap with **fast compile times** plus **good optimization quality**.

This is exactly the prompt: an Erlang T2 doing AOT-style optimizations
with profile-guided types and small-function inlining.

## 3. Maglev's IR: CFG-SSA, *not* sea-of-nodes

Quoted directly:
> "We decided to go with a somewhat traditional static single-assignment
> (SSA) based approach, using a CFG (control flow graph) rather than
> TurboFan's more flexible but cache unfriendly sea-of-nodes
> representation."

This is a deliberate reversal. After 10 years of TurboFan experience,
V8 chose CFG-SSA for the new fast optimizer. Reasons (inferred):
- Sea-of-nodes is hard to traverse efficiently — every traversal is a
  graph walk with cache misses.
- Standard SSA passes (CSE, GVN, DCE, allocation) have well-known
  efficient implementations on CFG-SSA.
- CFG-SSA matches the structure of bytecode/source code, making
  debugging easier.
- Linear register allocation works better on CFG-SSA.

Maglev IR has typed nodes, phi nodes for control flow merges, loop phis
for back-edges, and per-node "abstract interpreter frame state" for
deopt.

## 4. Type feedback consumed during graph construction

> "To be as fast as possible, Maglev does as much as possible at once.
> Instead of building a generic JavaScript graph and then lowering that
> during later optimization phases, which is a theoretically clean but
> computationally expensive approach, Maglev does as much as possible
> immediately during graph building."

This is the fundamental performance trick. Generic IR nodes never get
created; Maglev *starts* building specialised nodes:
- "If Maglev sees `o.x` and knows from the runtime feedback that `o`
  always has one specific shape, it will generate an SSA node to check
  at runtime that `o` still has the expected shape, followed by a cheap
  `LoadField` node."
- "Once a shape is confirmed, Maglev makes a side note that it now
  knows the shape of `o`, making it unnecessary to check the shape
  again later." → in-line type narrowing during construction.

Forms of feedback:
- **Checked feedback**: requires a runtime guard.
- **De-facto constants**: globals not yet mutated → loaded at compile
  time, baked into machine code; runtime mutations invalidate the
  blob.
- **Stable shape transitions**: register a dependency; if any object
  ever transitions away from this shape, invalidate.

## 5. Speculation and deopt

Maglev nodes that can deopt carry an **abstract interpreter frame
state** — a mapping from interpreter registers (Ignition's stack frame
slots) to SSA values. At codegen, this becomes deopt metadata used by
the runtime to reconstruct an Ignition frame on exit.

Maglev *reuses TurboFan's deoptimization runtime* — it generates
metadata in TurboFan's format. This is a huge implementation win:
TurboFan's deopt path is battle-tested; sharing it eliminates a whole
class of bugs.

> "This builds on the same deoptimization mechanism as used by
> TurboFan, allowing us to share most of the logic and take advantage of
> the testing of the existing system."

The lesson for us: **share deopt infrastructure with anything else that
needs it.** If we add OSR exit support to BeamAsm itself, T2 deopt should
piggy-back on the same metadata layout.

## 6. Register allocation: simple, fast, linear

Two-phase:
1. **Liveness prepass**: compute live ranges so registers can be freed
   when an SSA value is no longer needed; build the chain of uses.
2. **Allocation pass**: forward walk the graph with local greedy rules.
   - Keep values in their registers if possible.
   - On register pressure, spill the value not needed for longest.
   - On branch merges, merge abstract register states; emit "gap moves"
     to align values.

Spills are *retroactive* — when allocation determines a value must
spill, it tells the value to immediately spill on definition (right
after creation), and codegen emits the spill code. The definition
dominates all uses (SSA), so this is well-defined.

Frame layout simplification: rather than tracking which slots are
tagged on a per-instruction basis (TurboFan's approach), Maglev
**splits the stack frame into a tagged region and an untagged region**
and only stores the split point in deopt metadata. Vastly less
metadata, slight loss of layout flexibility.

## 7. Inlining

Not deeply documented in the public blog post. Comments in the V8
source indicate:
- Monomorphic call sites with small inline candidates inline.
- Inline depth and total node budget caps.
- Polymorphic sites use polymorphic inline caches; not inlined unless
  reduced to monomorphic by feedback.

## 8. OSR

Not deeply documented for Maglev specifically. TurboFan supports OSR;
Maglev presumably reuses similar mechanism for loop entry from
Sparkplug.

## 9. Production observability

Maglev shipped in Chrome M117 (Sept 2023). Energy use went *down*:
- JetStream: −3.5% energy
- Speedometer: −10% energy

Reason: less off-thread CPU spent compiling at TurboFan because Maglev
covers the gap, and Maglev's own compilation is cheap.

## 10. Lessons for an Erlang T2

This is the most directly relevant prior art. Concrete takeaways:

1. **CFG-SSA, not sea-of-nodes.** This is the considered design choice
   from a team that has lived with sea-of-nodes for ten years.
2. **Build specialised IR directly, don't lower a generic IR.** Type
   feedback consumed *during* graph construction, not in a separate
   lowering pass. Critical to compile-time performance.
3. **Side-table of known facts.** As Maglev walks the graph, it
   accumulates "known node info" — a side data structure that lets
   subsequent operations skip checks. Equivalent for Erlang: as we
   process SSA blocks, accumulate a `#{Var => RefinedType}` map that
   later operations consult.
4. **Single-pass, greedy linear register allocator.** Not graph
   coloring. Spill retroactively at definition.
5. **Tagged/untagged region split** for stack frame layout. We get this
   nearly for free in BEAM because Y registers are tagged Eterms by
   convention; new untagged spill region for unboxed integers/floats
   can sit alongside.
6. **Share deopt with the underlying tier.** The Maglev/TurboFan deopt
   metadata format is shared. We need a uniform "BEAM SSA position +
   X-register snapshot" deopt format used by both BeamAsm and T2.
7. **The middle tier wins on compile-time-amortised throughput.** Even
   if T2 produces only 30% better code than BeamAsm but compiles only
   3× slower, it covers the early-warm period that a heavy optimizer
   would miss entirely. There's no need to chase last-mile peak
   performance for the first version.
8. **Energy improvements** are a real production-deployment story. We
   should plan to measure energy/CPU-time-amortised metrics, not just
   single-benchmark speedup.
