# HiPE — High-Performance Erlang (Historical Native Compiler)

Sources:
- ["All you wanted to know about the HiPE compiler" — Sagonas et al., Erlang Workshop 2003](https://user.it.uu.se/~kostis/Papers/erlang03.pdf)
- ["HiPE on AMD64" — Luna & Pettersson, Erlang Workshop 2004](https://erlang.org/workshop/2004/amd64.pdf)
- ["HiPE overview slides" — Sagonas](https://user.it.uu.se/~kostis/Teaching/KT2-12/Slides/HiPE_overview.pdf)
- ["HiPErJiT: A Profile-Driven JIT for Erlang" — Kallas et al., IFL 2018](https://angelhof.github.io/files/papers/hiperjit-2018-ifl.pdf)
- ["The Road to the JIT" — erlang.org blog (2020)](https://www.erlang.org/blog/the-road-to-the-jit/)
- HiPE removal mailing-list thread, June 2020.

HiPE was the official Erlang native code compiler from approximately 2001
through OTP 23. It was deprecated in OTP 22 and removed in OTP 24. It is
critical prior art: same source language, same runtime, same data
representation, *much* deeper optimization than BeamAsm. Understanding
why HiPE didn't win is half the design constraint for T2.

## 1. Compilation pipeline

```
Erlang source
    ↓ (compiler)
BEAM bytecode
    ↓ (HiPE BEAM-to-Icode)
Icode      — register-based, infinite virtual registers, near-Erlang semantics
    ↓ (Icode optimizer)
Icode'
    ↓ (Icode-to-RTL)
RTL        — three-address, target-independent shape, target-specific operations
    ↓ (RTL optimizer)
RTL'
    ↓ (per-arch backend: x86, x86_64, sparc, ppc, arm)
machine code
```

Two intermediate representations, *separate* from BEAM SSA. HiPE was
written in Erlang and produced architecture-specific assembly via its
own backends (no LLVM, no asmjit).

### Icode

- Register-based (infinite virtual registers).
- Operations are Erlang-level: `call`, `enter`, `cons`, `mkfun`,
  `is_integer`, `add`.
- Stack discipline: virtual.
- Notable optimizations at this level: type analysis, function
  inlining, range analysis, bit-syntax matching, escape analysis.

### RTL

- Three-address code over typed registers.
- "Tagged registers" for proper Erlang values (carry tag bits) and
  "untagged registers" for raw machine values (e.g. unboxed floats,
  raw integers). Conversions between are explicit (`tag`, `untag`).
- Operations are closer to machine: explicit memory ops, address
  arithmetic, conditional moves.
- Optimizations: register allocation pre-prep, peephole, instruction
  scheduling, GC-test placement.

### Backend

- Architecture-specific assembler.
- Linear-scan register allocator (with optional iterative coloring on
  some platforms).
- Frame layout, GC stack maps, exception tables.

## 2. Mode mixing — the dual stack

HiPE's defining feature was **mode mixing**: a single Erlang process
could execute both BEAM-interpreted code and HiPE-compiled native code,
mixed across a single call chain.

To do this, each process had **two stacks**:
- A BEAM stack (the standard one) for interpreted frames.
- A *native stack* for compiled frames.

Calls between modes went through *mode-switch trampolines* with custom
return-address handling:
- `BEAM → native call`: trampoline copies arguments to the native
  calling convention, switches to the native stack, jumps to the
  callee.
- `native → BEAM call`: trampoline pushes a special return address
  that, when returned to, restores the BEAM stack and re-enters the
  interpreter.

The trampolines also had to handle exception propagation between the
two modes — `try`/`catch` had to find handlers on either stack.

## 3. Calling convention

HiPE x86_64 (per the AMD64 paper):
- `%rsi` ≈ HEAP_TOP (`HP`)
- `%rbp` ≈ process pointer (`P`)
- `%rsp` ≈ native stack
- Arguments in registers (`%rdi, %rdx, %rcx, %r8, %r9, %rax`) for the
  first 6, rest on stack.
- Reductions counter in a fixed register/memory location, decremented
  per call.

Notably: this is *different* from the BEAM calling convention used by
BeamAsm today. Native code generated by HiPE could not be called by
BeamAsm-generated code without a mode switch.

## 4. Garbage collection

GC participation required HiPE-emitted **GC stack maps**: at every safe
point (function call, allocation), HiPE recorded which native stack
slots/registers held tagged Erlang values vs raw integers. The GC walks
both stacks during collection, using BEAM stack maps for one half and
HiPE stack maps for the other.

Handling unboxed float arithmetic was a notable win — HiPE could keep
floats in FPU registers across a sequence of operations, boxing only
when crossing a function boundary or escaping to the heap.

## 5. Erlang-specific optimizations

The HiPE work delivered serious type-based optimizations, anticipating
much of what `beam_ssa_type` later did in the AOT compiler:
- Specialised arithmetic on small integers (no boxing fast paths).
- Unboxed float arithmetic across local sequences.
- Bit-syntax matching with state tracking (later subsumed by
  `beam_ssa_bsm`).
- Pattern-match compilation to decision trees.
- Inlining (intra-module, configurable depth).
- Range analysis to elide guards.

Notably, HiPE was an **AOT** compiler with no profile data. All its
optimization was based on what it could prove from source.

## 6. HiPErJiT (research, 2018)

Kallas et al. built **HiPErJiT** on top of HiPE: a runtime *profiling*
component identified hot modules, recompiled them with HiPE using
profile-guided hints (mostly inlining), and swapped the compiled
module in.

This is the closest published research analogue to what we want T2 to
do — profile-guided optimization on top of a heavyweight optimizer.
Worth re-reading the paper before designing the profiling subsystem;
in particular it documents what counters they collected and what
inlining decisions paid off.

## 7. Why HiPE was removed

From [The Road to the JIT](https://www.erlang.org/blog/the-road-to-the-jit/):

1. **Real workloads don't benefit much.** Big Erlang systems spend
   their time in message passing, ETS, GC, and BIF calls — none of
   which HiPE could optimize. Even 2–3× speedups on numeric inner loops
   moved end-to-end app perf only by single-digit percent.

2. **Cross-module optimization is impossible** because Erlang allows
   module reload. HiPE could not assume anything about another module's
   exports — defeating most type inference at module boundaries.

3. **The maintenance cost was unaffordable.** The OTP team had no
   internal Ericsson use; the only known production use was for
   speeding up Dialyzer. Carrying multi-architecture machine-code
   backends with custom assemblers, custom register allocators, custom
   GC stack maps, and dual-stack runtime support became unsustainable.

4. **Mode-mixing complexity blocked compiler evolution.** New BEAM
   instructions (e.g. for binary matching) and new compiler
   optimizations created bytecode patterns HiPE didn't handle — and
   adding HiPE support every time slowed down the rest of the
   compiler. By OTP 22, "HiPE was not fully functional — did not
   handle all BEAM instructions and combinations".

5. **Dual stacks broke interaction with NIF/port/dirty scheduler
   work.** Many runtime features had to special-case HiPE-vs-BEAM
   stack walks.

## 8. What we should *not* repeat

For T2 design, the explicit anti-patterns from HiPE are:

- **No dual stack.** T2 must execute on the same stack BeamAsm uses,
  with the same calling convention. This is non-negotiable: it's the
  single change that makes mode mixing free.
- **No separate machine-code backend.** asmjit (already used by
  BeamAsm) handles ARM64; reuse it. Don't add a new assembler.
- **No separate IR family.** HiPE had Icode + RTL on top of BEAM
  bytecode. The compiler now has BEAM SSA already. T2 should
  consume/produce BEAM SSA (with extensions), not invent a parallel IR.
- **No "compile every module" mode.** T2 must be selective — compile
  hot functions, leave the rest in BeamAsm.
- **No optimization that can't be invalidated by code reload.** Any
  speculation across module boundaries must register with the code
  purger and invalidate cleanly.
- **No GC-stackmap divergence.** Reuse BeamAsm's existing GC root
  discovery (X register array, stack walk). T2 frames must be
  GC-walkable using the same conventions.

## 9. What we should learn

- **Type-driven optimization works.** HiPE proved Erlang admits
  meaningful arithmetic and float specialisation. The AOT compiler
  picked this up; T2 picks up the part that needs runtime types.
- **Profile-guided inlining is the unique T2 win.** HiPErJiT showed
  this is implementable; it is the optimization the AOT compiler most
  cannot do.
- **Maintenance cost is the existential threat.** Plan T2 to be
  *small* (low five digits of source LOC max), to *reuse* existing
  infrastructure (BeamAsm assembler, beam_ssa_opt passes, beam_types
  lattice, asmjit), and to be possible to *turn off* if it stops
  paying its way.
